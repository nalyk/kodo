Development Ops: The Refined Change Pipeline

stateDiagram-v2

    pending --> triaging: brain classifies as dev

    triaging --> generating: issue needs code (codex exec)
    triaging --> auditing: PR ready for review (claude -p)
    triaging --> auto_merge: deps + CI green (zero LLM)

    state "hard-gates" as hard_gates
    note right of hard_gates

        ‚Ä¢ test suite must pass
        ‚Ä¢ semgrep / npm audit clean
        ‚Ä¢ diff ‚â§ max_diff_lines
        ‚Ä¢ scope limited to related files
    end note

    hard_gates --> auditing: all gates pass
    hard_gates --> deferred: any gate fails

    auditing --> scanning: claude confidence ‚â• 50
    auditing --> deferred: confidence < 50

    scanning --> auto_merge: confidence ‚â• 90 + scan clean
    scanning --> balloting: confidence 50-89
    scanning --> deferred: vulnerability found

    balloting --> guarded_merge: 2/3 consensus
    balloting --> deferred: no consensus

    auto_merge --> releasing: merge success
    guarded_merge --> releasing: CI green 48h window

    releasing --> resolved: auto-release (semver tag)
    releasing --> reverting: CI regression in 48h

    reverting --> deferred: revert merged

    deferred --> pending: new commits (max 2)
    deferred --> closed: auto-close with explanation

    closed --> [*]
    resolved --> [*]
Core principles: bash-enforced transitions, hard Layer 1 gates (Semgrep/tests/diff metrics), LLM as advisory Layer 2, principled 48h rollback window, --json-schema for structured confidence output. Auto-generated regression tests (Layer 1.5) catch behavioral changes existing tests miss. The releasing state auto-tags a semver release after merge if the repo config enables it ‚Äî feeding directly into the marketing engine's announcement flow.

7 Former Limitations ‚Äî Solutions Found

all addressed
1. "It will merge bad code sometimes"
Was: Hard gates + LLM review still miss semantic regressions not covered by tests.
Solution ‚Äî Auto-generated regression tests + behavioral diff assertions:
Before any code merge, add a Layer 1.5 between hard gates and LLM review. Codex generates targeted test cases specifically for the changed code paths: codex exec "Write 3-5 test cases that exercise the behavioral changes in this diff. Focus on edge cases and boundary conditions the existing tests don't cover." Run the generated tests. If they reveal unexpected behavior changes (test written expecting old behavior fails against new code), that's a real signal ‚Äî lower confidence by 20 points and route to balloting. Additionally, Claude writes behavioral diff assertions during review: "This change means function X now returns null instead of throwing when input is empty." Each assertion is checked against existing test coverage. Uncovered assertions get flagged in the review comment. This doesn't eliminate semantic risk ‚Äî but it catches a class of errors that neither static analysis nor existing tests catch: the code works differently, and nobody tested the difference. Residual risk: regressions that auto-generated tests also miss. This is the genuine inherent limit. The solution narrows the gap from "anything tests miss" to "things even an adversarial test generator misses."

2. "Marketing content is good enough, not great"
Was: Gemini produces generic, competent content without brand voice.
Solution ‚Äî Voice profile + few-shot examples + Claude editor pass:
Add a [voice] section to each repo's TOML: tone = "technical but approachable", vocabulary = ["contributors", "community", "ship"], anti_patterns = ["leverage", "synergy", "excited to announce"], personality = "a senior engineer who writes clearly and hates fluff". During auto-discovery, Claude scrapes the repo's existing release notes, README tone, and past announcement style, then stores 3-5 golden examples in repos/{name}.voice.md. Every marketing content generation call passes the voice profile + golden examples as few-shot context to Gemini. For high-stakes content (release announcements, contributor spotlights), add a Claude editor pass: Gemini drafts ‚Üí Claude rewrites with voice profile as instruction. One extra Claude call transforms generic content into content that sounds like the repo's existing voice. Add a community feedback loop: track GitHub Discussions reaction counts and reply rates per post. Monthly, feed engagement metrics back into the voice profile: "Posts with direct, technical tone got 3√ó more engagement than generic announcements. Adjust accordingly." Residual limit: voice consistency improves over time via the feedback loop but starts imperfect. The golden examples bootstrap quality from day one; the feedback loop refines it.

3. "PM analysis is pattern-matching, not product intuition"
Was: Claude analyzes GitHub data only. Cannot see outside the repo ‚Äî no market context, no user sentiment beyond issues.
Solution ‚Äî Domain knowledge file + external signal feeds + competitor tracking:
Each repo gets an optional KODO.md ‚Äî a one-time domain knowledge document written by the maintainer: product vision, target users, business model, strategic priorities, competitive landscape, domain glossary. Written once, updated occasionally. This is NOT per-PR human involvement ‚Äî it's project-level context that Claude reads automatically in every PM analysis. Not different from writing a README: you do it once and it informs everything. Add a [signals] section to repo TOML: RSS feeds for the domain, Stack Overflow tags, npm/PyPI download trends, competitor repo URLs. The scout checks these alongside GitHub: competitor_stars=$(gh api repos/{competitor}/--jq .stargazers_count), npm_downloads=$(curl -s "https://api.npmjs.org/downloads/point/last-week/{pkg}" | jq .downloads). The PM weekly report includes a competitive landscape section: "Competitor X released v3.0 last week (+400 stars). Their changelog mentions feature Y which aligns with our open issue #42." Add user sentiment mining: scrape GitHub Discussions, issue comments from non-team members, and Stack Overflow questions tagged with the project. Claude analyzes: "Users are repeatedly asking about feature Z but we have no issue for it." Residual limit: market intuition that exists nowhere in public data (private customer conversations, unspoken user frustration, strategic bets) remains inaccessible. The solution surfaces every signal that IS publicly available and currently ignored.

4. "GitHub-native, not universal"
Was: Only works with GitHub via gh CLI.
Solution ‚Äî Git provider adapter (kodo-git.sh) + validated alternatives:
The scout and brain never call gh directly. They call kodo-git.sh <action> <repo>, which reads provider from the repo TOML and routes to the correct CLI. GitHub: gh pr list, gh issue list, gh api ‚Äî full support. GitLab: glab (official GitLab CLI, open source, supports MR list, issue management, CI pipelines, releases). Validated: glab supports non-interactive auth, MR/issue operations, and CI management. Some commands still have interactive prompt quirks in edge cases, but core operations work headless. Gitea/Forgejo: REST API via curl + jq ‚Äî Gitea exposes PR, issue, release, and repo endpoints with OpenAPI docs. Bitbucket: REST API v2 via curl + jq. The adapter is ~50-80 lines per provider. The TOML gets: provider = "github" | "gitlab" | "gitea" | "bitbucket" plus api_base for self-hosted instances. Day-one support: GitHub + GitLab. Week-two: Gitea/Forgejo via API. Residual limit: provider-specific features (GitHub Discussions, GitLab epics) need per-provider SKILL.md extensions. The core dev/mkt/pm flows are provider-agnostic; advanced community features are not.

5. "Confidence scores are uncalibrated"
Was: Initial bands are guesses. No data until the system runs for a while.
Solution ‚Äî Bootstrap calibration + adaptive rolling window:
Phase 1 (shadow mode): During onboarding, replay the last 50 merged PRs through the confidence scoring system. For each: what would K≈çd≈ç have scored? Was that PR later reverted, hotfixed, or bug-reported? This gives a calibration dataset before going live. If the 90+ band shows a 15% incident rate on historical data, tighten to 95+ before launch. Phase 2 (live adaptive): Rolling 30-day window. Every merge outcome (clean, reverted, hotfixed) feeds the merge_outcomes table. An exponential moving average per band updates thresholds automatically: UPDATE confidence_bands SET threshold = threshold + 2 WHERE band = '90+' AND incident_rate_30d > 0.10. Bands tighten when incidents rise, loosen when the system proves accurate. No quarterly manual adjustment ‚Äî continuous, data-driven, automatic. Phase 3 (cross-repo calibration): Repos with similar profiles (same language, similar test coverage, similar CI maturity) share calibration data. A well-calibrated TypeScript repo's bands inform initial thresholds for a new TypeScript repo. Residual limit: calibration precision depends on volume. A repo with 2 PRs/month won't generate enough data for reliable band adjustment for months. Cross-repo pooling helps but doesn't fully solve low-volume repos.

6. "Claude down = degraded, survivable hours not days"
Was: Gemini fallback with capped confidence. Works but degrades decision quality.
Solution ‚Äî Queue-and-replay + Codex emergency reviewer + hibernate mode:
Tier 1 (0-30 min): Brain detects Claude failure (non-zero exit / timeout / rate-limit error in JSON).
